## Cache Format (zsCache / editor zscache)
- `0.idx` — Index file: 36-byte header + 36-byte entries (format `<QQIIIII` = hash, offset, orig_size, comp_size, seq, blocks, meta)
  - `meta & 0xFF` = compress_type (must be 10 for valid entries)  
  - `(meta >> 12) & 0xF` = dat file index
- `0-7.dat` — Data blob files (~1.38 GB each in editor cache)
- `fn0-7.1` / `FN0-7.1` — Filename hash mapping files (format `<QQI` entries: h1, h2, chain)
- **Compression**: LZHAM (compressType=10), NOT zlib. Decompressed via `lzham_x64.dll` (`lzham_z_uncompress`)
- **On-disk entry format**: First 20 bytes are header (checksum, constant 0x000AE7A4, sizes), actual LZHAM payload starts at byte 20
- **LZHAM DLL**: `seasun/editortool/qseasuneditor/seasunapp/httppacking/lzham_x64.dll`

## Lessons Learned
(Updated as mistakes are encountered)
- **Compression is LZHAM, NOT zlib/LZ4/LZMA**: Despite config files saying "compressType=10" and zlib being embedded in engine DLLs, the cache uses LZHAM. The game's `lzham_x64.dll` provides `lzham_z_uncompress` (zlib-compatible API).
- **Skip 20 bytes, not 16**: The on-disk entry header is 20 bytes (not 16 as initially assumed). The existing code in `pakv5_tex_extract.pyc` confirmed this.
- **Two cache locations exist**: `zsCache/dat/` (downloader, tiny ~0.1MB per FN file) vs `seasun/zscache/dat/` (editor, ~1.38GB per .dat file). Only the editor cache has substantial data.
- **False-positive TGA detection**: When decompression produces garbage, random bytes can match TGA header patterns. Always verify with LZHAM first.
- **FN*.1 files don't contain readable filenames**: They contain hash-to-hash mappings (h1→h2→chain), not path strings.
- **The 1,231 DDS files in `seasun/client/data/` are for OTHER maps** (dungeons/秘境), not the target outdoor map.
- **Dat file naming differs**: Client uses `FN{i}.1`, editor uses `{i}.dat`
- **Python 3.10 needed for decompiling .pyc**: Bytecode version mismatch prevents 3.11 from loading 3.10 .pyc files

## Hash Function Cracking — Complete Guide

### The Problem
The zsCache stores files by hash — no readable filenames. To extract a specific file (e.g., a texture), you need to **compute its hash from the game path**.

### Hash Architecture
Each file has TWO hashes:
- **h1** (IDX hash): 60-bit. Used as the key in `0.idx`. **STILL UNKNOWN** — bypassed via reverse-lookup.
- **h2** (FN hash): 64-bit composite. Used in `fn{0-7}.1` to link h2→h1.
  - `h2 = (dir_hash << 40) | file_hash`
  - **dir_hash** (top 24 bits, 22-bit effective): DJB2 of parent directory
  - **file_hash** (bottom 40 bits): xxHash64 of full path

### Exact Formulas
```python
# dir_hash: DJB2 with init=5381, masked to 22 bits
def djb2(data_bytes, init=5381, mask=0x3FFFFF):
    h = init
    for b in data_bytes:
        h = (h * 33 + b) & mask
    return h

# file_hash: xxHash64 with seed=0, masked to 40 bits
file_hash = xxhash.xxh64(data_bytes, seed=0).intdigest() & 0xFFFFFFFFFF

# The combined h2
h2 = (dir_hash << 40) | file_hash
```

### Critical: Encoding is GBK
- ALL path hashes use **GBK-encoded bytes**, NOT UTF-8
- For ASCII-only paths, GBK and UTF-8 produce identical bytes
- For Chinese character filenames (most game assets), **GBK is required**
- Paths must be **lowercased** and use **forward slashes** before encoding

### Double-Encoding Trap
Some pipeline steps (e.g., texture_needs.json built from UTF-8 JsonInspack) produce **garbled strings** where GBK bytes were misread as UTF-8:
```python
# Recovery: encode back to UTF-8 → gives original GBK bytes → decode as GBK
real_name = garbled_string.encode('utf-8').decode('gbk')
```

### Extension Swapping
Material files reference `.tga` but the cache stores `.dds` versions:
- Always try both extensions when computing h2
- 751 of 753 extracted textures are DDS; only 2 are actual TGA

### Reverse-Lookup Strategy (bypassing h1)
Since h1 is unknown, we compute h2 → look it up in FN files → get h1 → look up IDX:
```
known_path → compute h2 → FN[h2] → h1 → IDX[h1] → (offset, size, dat_file)
```

### What Failed (DO NOT REPEAT)
1. **All standard hash functions tested against h1** — ZERO matches with MD5, SHA-1, SHA-256, CRC32, FNV-1a, MurmurHash, CityHash, SipHash at ALL seeds/offsets/masks
2. **Brute-force algebraic solver for DJB2 init** — tested W=24,32,48 bit widths before finding W=22
3. **UTF-8 encoding for Chinese paths** — produces wrong hashes; only GBK works
4. **zlib decompression** — the cache uses LZHAM even though engine DLLs embed zlib
5. **CDN file lists for direct path matching** — filenames exist but hash computation is the bottleneck

### What Worked (The Breakthroughs, In Order)
1. **CDN batch download** (g_OpenFile) → got 36 (path, h1) pairs via content matching
2. **h2 composite structure** discovered: dirs share upper bits, files differ in lower bits
3. **CachedShaders analysis** → proved multiplier=33 (DJB2-like)
4. **Per-directory init diagnostic** → all share bottom 22 bits = 5381 (standard DJB2 init!)
5. **DJB2 with init=5381 & 0x3FFFFF** → 20/21 dirs match (22-bit effective width)
6. **xxHash64 seed=0 low40** → 34/36 file hashes match
7. **GBK encoding test** → `utf-8: NOT found`, `gbk: FOUND` for Chinese filenames
8. **Double-encoding recovery** → `garbled.encode('utf-8').decode('gbk')` for the 9 remaining paths

### Full Extraction Pipeline
```bash
# Step 1: Compute h2 for target file
h2 = compute_h2("data/source/maps_source/texture/some_texture_01h.dds")

# Step 2: Look up h1 in FN files
h1 = fn_map[h2]  # from fn{0-7}.1

# Step 3: Look up IDX entry
entry = idx[h1]  # gives offset, sizes, dat_file_index

# Step 4: Read from dat file and decompress
data = decompress_lzham(dat_files[entry.dat_idx], entry.offset, entry.comp_size, entry.orig_size)
```

## Phase 3 Summary (COMPLETED — Multi-Subset Meshes + Face Winding)

### Multi-Subset Mesh Format
- JX3 meshes contain **multiple subsets** (sub-meshes), each with its own material/texture
- JsonInspack format: `lods[0].Group[0].Subset[i]` — **only LOD 0, Group 0** is relevant
- Prior bug: code read ALL 4 groups × 5 subsets = 20 entries instead of just 5
- Each subset has independent `BaseColorMap`, `Map(MRE)`, `NormalTexture` parameters
- GLB output: one **primitive per subset**, each with its own material
- `texture-map.json`: per-mesh array of subset entries `[{albedo, mre, normal}, ...]`

### Face Winding (100% Wrong)
- JX3 engine uses **left-handed** (DirectX) coordinate system
- glTF / Three.js uses **right-handed** coordinate system
- **ALL faces were back-facing** until fixed
- Fix: negate Z vertex positions + reverse triangle winding order in `mesh_to_glb()`
- Also applies to normals: negate Z component

### MRE Channel Mapping Fix
- Game stores MRE as `R=metallic, G=roughness, B=emissive`
- Three.js `MeshStandardMaterial` expects `R=roughness, G=metallic` (reversed!)
- Fix: swap R↔G channels during DDS→PNG conversion
- flipY was also wrong for DDS textures — needed to flip vertically

### World-Location Mirroring (LH→RH Coordinate Conversion)
- **Problem**: Mesh vertices were Z-negated (LH→RH) but entity world positions and terrain were still in LH
- **Root cause**: Entity transform matrices stored in LH row-major format, never converted to RH
- **Entity transform fix** (`entities.js`):
  - Position extraction: `z: -m[14]` (negate Z translation)
  - Full matrix: `M_rh = S * M_lh * S` where `S = diag(1,1,-1,1)`
  - Negates: `m[8], m[9]` (row 0,1 col 2), `m[2], m[6]` (row 2 col 0,1), `m[14]` (Z translation), `m[11]` (row 2 col 3 perspective)
- **Terrain fix** (`terrain.js`):
  - Vertex Z negated: `positions[vi * 3 + 2] = -worldZ`
  - Triangle winding reversed to match negated Z
  - LOD center Z negated
  - World bounds Z negated with min/max swapped
  - `getHeightAt()`: converts incoming RH worldZ back to LH for heightmap lookup
- **App/UI fixes** (`app.js`):
  - Config-based start position: Z negated
  - Minimap marker: removed `(1-nz)` inversion (Z negation already handles it)
  - Region display: converts camera RH Z back to LH for region calculation
- **Entity matrix format**: Row-major 4×4 in JSON, position at indices `[12, 13, 14]`
  ```
  Row-major layout:
  m[0]  m[1]  m[2]  m[3]     ← row 0 (right vector)
  m[4]  m[5]  m[6]  m[7]     ← row 1 (up vector)
  m[8]  m[9]  m[10] m[11]    ← row 2 (forward vector)
  m[12] m[13] m[14] m[15]    ← row 3 (translation)
  ```
  Three.js uses column-major, so when calling `mat4.set()`, transpose by reading columns:
  `mat4.set(m[0], m[4], m[8], m[12], m[1], m[5], m[9], m[13], ...)`

### Lessons Learned (Phase 4)
- When converting coordinate systems, EVERY component that uses world coordinates must be updated:
  entities, terrain, camera start position, minimap, height queries, region display
- Entity matrices need the full `S*M*S` reflection, not just Z position negation
- Terrain winding must be reversed when Z is negated (otherwise terrain faces backward)
- `getHeightAt()` must convert RH→LH internally since heightmap data is in LH coordinates
- Missing textures may be genuinely unavailable — implement fallback rather than assuming extraction failure
